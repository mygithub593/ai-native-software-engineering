# چارچوب آموزش «مهندسی نرم‌افزار در عصر AI»

این چارچوب بر یک فرض بنیادین بنا شده است:

**نقش انسان از “نویسنده کد” به “طراح سیستم، تحلیل‌گر مسئله و هدایت‌کننده AI” تغییر کرده است.**

پس آموزش هم باید از **سینتکس‌محوری** به **سیستم‌محوری** منتقل شود.

---

## 1) اصول حاکم بر کل مسیر

این‌ها ستون‌های اصلی مدل آموزشی هستند:

### اصل 1: مسئله قبل از ابزار

اول باید بفهمی چه چیزی را می‌خواهی بسازی، بعد تصمیم بگیری با چه زبان و فریم‌ورکی.

### اصل 2: معماری قبل از پیاده‌سازی

پیش از نوشتن هر خط کد باید ساختار سیستم، مرزها، داده‌ها و جریان‌ها روشن باشند.

### اصل 3: کدنویسی به‌عنوان مهارت مکمل

هدف، حفظ کردن همه جزئیات زبان نیست؛ هدف این است که کد را بخوانی، ارزیابی کنی و اصلاح کنی.

### اصل 4: AI ابزار اصلی تولید است، نه منبع اصلی فهم

AI می‌تواند کد بسازد، اما فهم مسئله، تشخیص trade-offها و کنترل کیفیت هنوز با انسان است.

### اصل 5: یادگیری پروژه‌محور

یادگیری باید از همان ابتدا حول یک یا چند مسئله واقعی باشد، نه حول تمرین‌های انتزاعی.

### اصل 6: خروجی‌محور بودن

هر مرحله باید خروجی قابل مشاهده داشته باشد: سند، دیاگرام، معماری، API spec، تست، یا یک ماژول واقعی.

---

# 2) تعریف دقیق هدف آموزشی

هدف این دوره این نیست که تو تبدیل به یک «برنامه‌نویس سنتی تمام‌وقت» شوی.

هدف این است که تو به یک:

**AI-Native Software Engineer**

تبدیل شوی؛ یعنی فردی که می‌تواند:

* مسئله را به سیستم تبدیل کند
* معماری طراحی کند
* AI را برای تولید و تغییر کد هدایت کند
* خروجی را ارزیابی کند
* کیفیت، امنیت و مقیاس‌پذیری را کنترل کند
* یک محصول واقعی را از ایده به اجرا برساند

---

# 3) ساختار نهایی آموزش

من این مسیر را به 6 لایه اصلی تقسیم می‌کنم:

---

## لایه 0: نقشه ذهنی مهندس نرم‌افزار

این لایه پیش‌نیاز ورود به هر چیز دیگری است.

### هدف

تغییر زاویه دید از “کدنویسی” به “ساخت سیستم”.

### خروجی

تو بتوانی برای هر ایده توضیح بدهی:

* مسئله چیست؟
* کاربر کیست؟
* ارزش محصول چیست؟
* چه چیزی باید خودکار شود؟
* چه چیزی باید قابل اعتماد، امن و مقیاس‌پذیر باشد؟

### مفاهیم

* Software as a System
* Problem Space vs Solution Space
* Product Thinking
* Technical Thinking
* Trade-off thinking

---

## لایه 1: تحلیل مسئله و مدل‌سازی محصول

این لایه مهم‌ترین نقطه شروع است.

### هدف

هر ایده را به زبان قابل طراحی و پیاده‌سازی تبدیل کنی.

### مفاهیم

* Requirement
* Use Case
* User Story
* Persona
* Functional Requirement
* Non-functional Requirement
* Business Rule
* Workflow
* Constraint
* Acceptance Criteria

### خروجی

برای هر ایده بتوانی یک **Problem Statement** و یک **Feature Breakdown** بنویسی.

### مثال خروجی

برای یک پلتفرم مدیریت پروژه:

* کاربران چه کسانی‌اند؟
* چه داده‌هایی ثبت می‌شود؟
* چه فرایندهایی وجود دارد؟
* چه خروجی‌هایی لازم است؟
* چه محدودیت‌هایی داریم؟
* کیفیت مطلوب چیست؟

---

## لایه 2: تفکر سیستمی و معماری

اینجا وارد قلب مهندسی نرم‌افزار می‌شوی.

### هدف

فهم ساختار سیستم و اجزای آن.

### مفاهیم

* Client / Server
* Frontend / Backend
* API
* Database
* Cache
* Authentication / Authorization
* Storage
* Queue / Event
* State
* Sync vs Async
* Monolith vs Modular Monolith vs Microservices

### خروجی

بتوانی سیستم را به اجزای منطقی تقسیم کنی و مرزهای هر جزء را مشخص کنی.

### مهارت کلیدی

توانایی پاسخ دادن به این سؤال:
**“این قابلیت باید کجا زندگی کند و چرا؟”**

---

## لایه 3: معماری نرم‌افزار و طراحی

این لایه برای تصمیم‌گیری فنی است.

### هدف

انتخاب ساختار مناسب برای پیاده‌سازی.

### مفاهیم

* Layered Architecture
* Clean Architecture
* Hexagonal Architecture
* Domain Driven Design (در حد کاربردی)
* Modular Design
* Design Patterns
* Separation of Concerns
* Dependency Management
* Interface Design

### خروجی

تو باید بتوانی:

* معماری پیشنهادی برای هر پروژه را ترسیم کنی
* دلیل انتخاب معماری را توضیح بدهی
* از overengineering جلوگیری کنی
* برای رشد آینده سیستم، مسیر طراحی داشته باشی

### نکته مهم

اینجا هدف یاد گرفتن اسم معماری‌ها نیست؛ هدف **توانایی انتخاب معماری مناسب** است.

---

## لایه 4: همکاری حرفه‌ای با AI

این بخش، تفاوت اصلی مدل جدید با آموزش سنتی است.

### هدف

یاد بگیری AI را مثل یک تیم توسعه هدایت کنی.

### مفاهیم

* Prompting برای توسعه نرم‌افزار
* Specification-driven development
* تقسیم کار برای AI
* AI-assisted debugging
* AI-assisted refactoring
* AI-assisted testing
* Code review with AI
* Document generation with AI

### خروجی

تو بتوانی:

* یک سند فنی روشن برای AI بنویسی
* خروجی AI را به اجزای کنترل‌پذیر تقسیم کنی
* خطاهای AI را تشخیص بدهی
* از AI برای تسریع، نه برای کورکورانه‌سازی، استفاده کنی

### قاعده طلایی

AI باید **تولیدکننده** باشد، اما انسان باید **معمار، داور و مسئول کیفیت** بماند.

---

## لایه 5: سواد کدنویسی

اینجا برخلاف مسیر سنتی، کدنویسی هدف نهایی نیست؛ ابزار کنترل و اصلاح است.

### هدف

توانایی خواندن، فهمیدن و تغییر دادن کدی که AI یا دیگران تولید کرده‌اند.

### مفاهیم حداقلی

* Variables / Data Types
* Functions
* Classes
* Modules / Packages
* Error Handling
* JSON
* HTTP
* CRUD
* Database access
* Logging
* Configuration
* Basic async concepts

### خروجی

تو باید بتوانی:

* کد را بخوانی
* منطق آن را بفهمی
* باگ‌ها را پیدا کنی
* تغییرات دقیق انجام بدهی
* بخش‌هایی را دوباره طراحی کنی

### نکته مهم

اینجا نیازی به حافظه‌محوری افراطی نیست. تسلط کافی بر خواندن و اصلاح مهم‌تر از حفظ ریزجزئیات است.

---

## لایه 6: کیفیت مهندسی، تست، امنیت و استقرار

این لایه برای حرفه‌ای شدن ضروری است.

### هدف

ساخت سیستم قابل اعتماد، نه فقط کد کارا.

### مفاهیم

* Unit Test
* Integration Test
* End-to-End Test
* Testability
* Logging / Monitoring
* Error Tracking
* Security Basics
* Validation
* Input Sanitization
* Version Control
* Deployment Basics
* CI/CD مفهومی

### خروجی

تو بتوانی یک سیستم را:

* تست کنی
* خطاهایش را پایش کنی
* از نظر امنیتی حداقلی محافظت کنی
* به محیط اجرا منتقل کنی

---

# 4) چیزهایی که عمداً در ابتدای مسیر عقب می‌افتند

در این چارچوب، چند موضوع عمداً در اولویت پایین قرار می‌گیرند:

* حفظ کردن کامل سینتکس
* ورود زودهنگام به چند زبان مختلف
* رفتن مستقیم به فریم‌ورک‌ها
* الگوریتم‌های سنگین و رقابتی
* جزئیات deep داخلی زبان‌ها
* معماری‌های بیش‌ازحد پیچیده در شروع

این‌ها حذف نشده‌اند؛ فقط **زمان ورودشان عوض شده است**.

---

# 5) ترتیب صحیح یادگیری در این مدل

ترتیب نهایی من این است:

1. **ذهنیت مهندسی نرم‌افزار**
2. **تحلیل مسئله و مدل‌سازی محصول**
3. **تفکر سیستمی**
4. **معماری نرم‌افزار**
5. **همکاری مؤثر با AI**
6. **سواد کدنویسی حداقلی و کاربردی**
7. **تست، کیفیت، امنیت، استقرار**
8. **ساخت پروژه واقعی**

این ترتیب عمداً از “فهم” به “ساخت” می‌رود، نه از “سینتکس” به “فریم‌ورک”.

---

# 6) معیار موفقیت این چارچوب

اگر این مدل درست اجرا شود، خروجی نهایی باید این باشد که تو بتوانی:

* یک ایده را به سند فنی تبدیل کنی
* برای آن معماری انتخاب کنی
* با AI ماژول‌ها را تولید کنی
* کد را بخوانی و اصلاح کنی
* تست بنویسی یا تولید کنی
* محصول را به نسخه قابل استفاده برسانی

یعنی از یک مصرف‌کننده ابزار، به یک **طراح و هدایت‌گر سیستم** تبدیل شوی.

---

# 7) جمع‌بندی نهایی

پس چارچوب نهایی چنین است:

**آموزش مهندسی نرم‌افزار در عصر AI باید:**

* مسئله‌محور باشد
* معماری‌محور باشد
* پروژه‌محور باشد
* AI-محور باشد
* کدنویسی‌محورِ کلاسیک نباشد

و هویت حرفه‌ای هدف این مسیر هم این است:

**AI-Native Software Engineer**

نه صرفاً یک programer سنتی، و نه صرفاً یک کاربر ابزارهای AI.
