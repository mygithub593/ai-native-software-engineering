# 02.02 — Requirements vs Features

---

## اهداف یادگیری

در پایان این درس باید بتوانی:

* تفاوت Requirement و Feature را توضیح دهی.
* نیازمندی‌های واقعی را از راه‌حل‌ها جدا کنی.
* برای یک محصول فهرست نیازمندی تهیه کنی.
* از اشتباه رایج «پریدن مستقیم به Featureها» جلوگیری کنی.

---

# چرا این درس مهم است؟

بسیاری از تیم‌های نرم‌افزاری از روز اول درباره این موضوعات صحبت می‌کنند:

* داشبورد
* اپلیکیشن موبایل
* چت
* هوش مصنوعی
* گزارش‌گیری
* اعلان‌ها

اما تقریباً هیچ‌کس نمی‌پرسد:

> چرا اصلاً این قابلیت‌ها لازم هستند؟

به همین دلیل پروژه‌ها پر از قابلیت می‌شوند اما مشکل اصلی کاربر را حل نمی‌کنند.

---

# Requirement چیست؟

Requirement یعنی:

> چیزی که سیستم باید برای حل مسئله فراهم کند.

تمرکز Requirement روی **نیاز** است.

نه روی ظاهر راه‌حل.

---

مثال:

```text
مدیر پروژه باید بتواند آخرین وضعیت پروژه را مشاهده کند.
```

این یک Requirement است.

چون بیان می‌کند کاربر به چه چیزی نیاز دارد.

---

مثال دیگر:

```text
کاربر باید بتواند مستندات پروژه را جستجو کند.
```

باز هم Requirement است.

---

# Feature چیست؟

Feature یعنی:

> قابلیتی که برای برآورده کردن یک Requirement ساخته می‌شود.

تمرکز Feature روی پیاده‌سازی است.

---

مثال:

```text
داشبورد مدیریتی
```

این Feature است.

---

مثال:

```text
جستجوی هوشمند مبتنی بر AI
```

این Feature است.

---

مثال:

```text
اعلان ایمیلی
```

این Feature است.

---

# رابطه Requirement و Feature

```text
Problem
↓
Requirement
↓
Feature
↓
Implementation
```

این یکی از مهم‌ترین نمودارهای کل دوره است.

---

مثال:

مسئله:

```text
اطلاعات پروژه پراکنده است.
```

---

Requirement:

```text
کاربر باید بتواند اطلاعات پروژه را به سرعت پیدا کند.
```

---

Feature:

```text
جستجوی سراسری
```

یا:

```text
AI Search Assistant
```

---

دقت کن:

برای یک Requirement ممکن است چند Feature مختلف وجود داشته باشد.

---

# اشتباه رایج شماره ۱

مشتری می‌گوید:

```text
ما یک داشبورد می‌خواهیم.
```

بسیاری از توسعه‌دهندگان این را Requirement در نظر می‌گیرند.

اما این یک Feature است.

باید بپرسیم:

> چرا داشبورد می‌خواهید؟

پاسخ:

```text
برای مشاهده وضعیت لحظه‌ای پروژه‌ها
```

حالا Requirement را پیدا کرده‌ایم.

---

# اشتباه رایج شماره ۲

مشتری می‌گوید:

```text
ما به AI نیاز داریم.
```

این Requirement نیست.

این Feature هم نیست.

این صرفاً یک راه‌حل پیشنهادی است.

باید بپرسیم:

```text
AI قرار است چه مشکلی را حل کند؟
```

مثلاً:

```text
تهیه گزارش پروژه زمان زیادی می‌گیرد.
```

این Requirement را آشکار می‌کند.

---

# Functional Requirement

این نوع نیازمندی مشخص می‌کند:

> سیستم چه کاری باید انجام دهد؟

مثال:

```text
کاربر باید بتواند پروژه جدید ایجاد کند.
```

---

مثال:

```text
کاربر باید بتواند فایل بارگذاری کند.
```

---

مثال:

```text
کاربر باید بتواند گزارش دریافت کند.
```

---

# Non-Functional Requirement

این نوع نیازمندی مشخص می‌کند:

> سیستم چگونه باید عمل کند؟

---

مثال:

```text
زمان پاسخ کمتر از ۲ ثانیه باشد.
```

---

مثال:

```text
سیستم ۹۹.۹٪ در دسترس باشد.
```

---

مثال:

```text
اطلاعات کاربران رمزنگاری شود.
```

---

بسیاری از پروژه‌ها روی Functional Requirements تمرکز می‌کنند و Non-Functional Requirements را فراموش می‌کنند.

---

# مثال روی پروژه خودمان

پلتفرم مدیریت پروژه مبتنی بر AI

---

Requirement 1

```text
مدیر پروژه باید بتواند آخرین وضعیت پروژه را مشاهده کند.
```

Featureهای ممکن:

```text
Dashboard

Progress Widget

Project Summary View
```

---

Requirement 2

```text
کاربر باید بتواند اطلاعات پروژه را سریع پیدا کند.
```

Featureهای ممکن:

```text
Global Search

AI Search

Document Indexing
```

---

Requirement 3

```text
تهیه گزارش پیشرفت پروژه باید سریع‌تر شود.
```

Featureهای ممکن:

```text
AI Report Generator

Auto Summary

Weekly Report Builder
```

---

# نگاه AI-Native

یکی از خطرات عصر AI این است:

AI می‌تواند در چند دقیقه ده‌ها Feature پیشنهاد دهد.

اما AI معمولاً نمی‌پرسد:

```text
آیا این Feature واقعاً یک Requirement را پوشش می‌دهد؟
```

وظیفه مهندس نرم‌افزار این است که:

هر Feature را به یک Requirement متصل کند.

اگر نتوانستی این اتصال را پیدا کنی:

احتمالاً آن Feature ضروری نیست.

---

# قانون طلایی

قبل از ساخت هر Feature بپرس:

> این قابلیت دقیقاً کدام Requirement را پوشش می‌دهد؟

اگر جواب روشنی وجود نداشت:

فعلاً آن قابلیت را نساز.

---

# خلاصه درس

* Requirement نیاز واقعی کاربر است.
* Feature راهی برای برآورده کردن آن نیاز است.
* یک Requirement می‌تواند چند Feature مختلف داشته باشد.
* بسیاری از افراد Feature را با Requirement اشتباه می‌گیرند.
* مهندسی نرم‌افزار خوب از Requirement شروع می‌شود نه از Feature.
